## 프로젝트 개요

이 프로젝트는 "Honeypot Integrated Platform"이라는 캡스톤 디자인 프로젝트입니다.
다중 허니팟 + ML 공격 분류 + LLM 분석 + 공격 시나리오 자동화를 하나의 통합 대시보드로 묶었습니다.
현재는 로컬 Docker 기반 구조에서 AWS T-Pot HIVE 연동으로 확장하는 시점입니다.

GitHub: https://github.com/Donghyun0918/honeypot-integrated


## 기술 스택

- Frontend : Next.js 15 (App Router, TypeScript, "use client")
- Backend  : FastAPI (Python, async, JWT 인증, SQLAlchemy + SQLite)
- ML       : scikit-learn RandomForest (binary + multi-class 분류)
- LLM      : Ollama (gemma3 또는 llama3.1:8b) — 고위험 공격 자연어 분석
- Profiling: Spring Boot 3 (별도 리포 ../profiling-service, PostgreSQL 16)
- 허니팟   : Cowrie, Heralding, OpenCanary, SNARE/Tanner, Dionaea, Mailoney, Conpot
- 공격자   : Kali Linux 컨테이너 (nmap, hydra, sqlmap, nc, curl, sshpass)
- 인프라   : Docker Compose (14개 컨테이너), AWS EC2 t3a.xlarge (T-Pot용)


## 시스템 구조 (기존 — 로컬 Docker)

[브라우저 :8001]
    └─► Next.js Frontend
            ├─► FastAPI Backend (:8000)
            │       ├── Docker SDK → 유저별 허니팟 컨테이너 생성/삭제
            │       ├── Kali (docker exec) → 공격 시나리오 9가지 실행
            │       ├── parse_logs.py → dataset.csv (7종 허니팟 로그 파싱)
            │       ├── ml_prepare.py → dataset_ml.csv (라벨링 + 피처)
            │       ├── ml_service.py → RandomForest 학습/추론, MITRE ATT&CK 점수
            │       └── Ollama (:11434) → MITRE ≥ 70인 공격만 LLM 분석
            └─► Spring Boot Profiling (:8090)
                    └── PostgreSQL 16 (공격 이력 영속화)

유저별 허니팟 컨테이너 이름 패턴: hp_{username}_{honeypot}
로그 마운트 경로: ${HONEYPOT_LOGS_HOST}/{username}/{honeypot}/
ML 모델 저장 경로: /app/data/ml_models/binary_model.pkl, multi_model.pkl


## 데이터 흐름 (한 사이클)

1. 시나리오 실행: Frontend → POST /api/scenarios/{id}/run → Backend → Kali → 허니팟
2. 로그 수집: 허니팟 → /honeypot_logs/{user}/{honeypot}/
3. 데이터셋 생성: POST /api/dataset/generate → parse_logs.py → dataset.csv
                  POST /api/ml/prepare/run → ml_prepare.py → dataset_ml.csv
4. 모델 학습: POST /api/ml/train → RandomForest → *.pkl 저장
5. 분류+분석: classify() → MITRE 점수 → Ollama → riskScore/severity/summary/solution
              → Spring Boot 프로파일링 저장

## 알려진 이슈 및 해결 내역

1. parse_logs.py exit code 255
   → 스크립트 끝에 sys.exit(0) 추가
   → 엔드포인트에서 exit code 대신 CSV 파일 존재 여부로 성공 판단

2. ML 페이지 "TypeError: Failed to fetch"
   → localhost가 IPv6(::1)로 해석되어 Docker IPv4와 충돌
   → frontend 전체에서 localhost → 127.0.0.1 교체

3. CSV 다운로드 401 오류
   → <a href> 직접 링크는 JWT 미전달
   → fetch + createObjectURL 방식으로 교체

4. WSL에서 chmod 적용 안 됨
   → /mnt/d/는 NTFS라 chmod 무시됨
   → WSL 홈(~/)으로 파일 복사 후 chmod 적용

## 개발 환경

OS: Windows 11 + WSL2 (Ubuntu)
Docker: Docker Desktop (WSL2 백엔드)
경로 규칙: .env의 호스트 경로는 /mnt/d/... 형식 사용 (WSL 기준)
주의: Windows 경로(D:/...)와 WSL 경로(/mnt/d/...)를 섞으면 컨테이너가 다른 파일시스템 뷰를 봄

## 코딩 컨벤션

Python:
  - PEP8, 타입 힌트 권장
  - 비동기 라우터는 async def
  - 블로킹 작업은 run_in_executor 또는 threading.Thread(daemon=True)

TypeScript:
  - Next.js 15 App Router
  - 클라이언트 컴포넌트는 "use client" 명시
  - 토큰은 getToken()으로 일관되게 접근
  - API 호출 시 127.0.0.1:8000 사용 (localhost 금지 — IPv6 충돌)

Bash 시나리오:
  - set -e 필수
  - 모든 IP/포트는 환경변수로 주입 (하드코딩 금지)
